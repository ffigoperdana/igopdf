# IGO - Rencana Implementasi Lengkap

## Ringkasan Eksekutif

Proyek ini mengubah BentoPDF (PDF toolkit client-side open source) menjadi **igo** — aplikasi web internal self-hosted untuk penggunaan PDF perusahaan. BentoPDF hanya akan menjadi mesin PDF processing, sedangkan semua UI, branding, autentikasi, dan manajemen pengguna harus dibangun ulang.

**Arsitektur Final**: Frontend (Vite + Tailwind) + Backend API (Node.js/Express) + PostgreSQL + Nginx + Docker Compose

**Lisensi**: AGPL-3.0 (atribusi BentoPDF harus dipertahankan secara legal)

## 2. Arsitektur

### 2.1 Arsitektur Final

```
┌─────────────┐     ┌──────────────────┐     ┌────────────┐
│   Nginx     │────▶│  Frontend (SPA)  │     │ PostgreSQL │
│  (reverse   │     │  Vite + Tailwind │     │            │
│   proxy)    │     │  Static files    │     │            │
│             │────▶│                  │     └────────────┘
│   :80/:443  │     └──────────────────┘            ▲
│             │                                     │
│             │────▶┌──────────────────┐            │
│             │     │  Backend API     │────────────┘
└─────────────┘     │  Node.js/Express │
                    │  :3000           │
                    └──────────────────┘
```

### 2.2 Flow Autentikasi

1. User mengakses `/` → redirect ke `/login` jika tidak ada session
2. Login form: username + password + CAPTCHA
3. Backend validasi credentials → buat session (HttpOnly cookie)
4. Session timeout: 2-3 jam
5. Refresh → session persist (cookie-based)
6. Logout → invalidate session + clear cookie

### 2.3 Flow RBAC

- `admin`: Full access (dashboard, user management, all tools)
- `user`: Limited access (home, tools, profile, change password)

### 2.4 CAPTCHA Architecture

Backend-generated CAPTCHA:
1. Backend generate CAPTCHA image + token
2. Token disimpan di cache dengan TTL singkat (5 menit)
3. User submit CAPTCHA response + token
4. Backend verifikasi: token valid + response benar + belum dipakai (one-time) + rate limit
5. Frontend-only validation TIDAK bisa dipercaya

## 8. Schema Database

### 8.1 Users Table

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(20) NOT NULL DEFAULT 'user' CHECK (role IN ('admin', 'user')),
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    last_login TIMESTAMP WITH TIME ZONE,
    created_by UUID REFERENCES users(id)
);
```

### 8.2 Sessions Table

```sql
CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash VARCHAR(255) NOT NULL UNIQUE,
    ip_address INET,
    user_agent TEXT,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_sessions_user_id ON sessions(user_id);
CREATE INDEX idx_sessions_token_hash ON sessions(token_hash);
CREATE INDEX idx_sessions_expires_at ON sessions(expires_at);
```

### 8.3 Login Attempts Table

```sql
CREATE TABLE login_attempts (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    ip_address INET NOT NULL,
    success BOOLEAN NOT NULL,
    attempted_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_login_attempts_username ON login_attempts(username);
CREATE INDEX idx_login_attempts_ip ON login_attempts(ip_address);
CREATE INDEX idx_login_attempts_time ON login_attempts(attempted_at);
```

### 8.4 Audit Logs Table

```sql
CREATE TABLE audit_logs (
    id BIGSERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    action VARCHAR(100) NOT NULL,
    details JSONB,
    ip_address INET,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_audit_logs_user_id ON audit_logs(user_id);
CREATE INDEX idx_audit_logs_action ON audit_logs(action);
CREATE INDEX idx_audit_logs_created_at ON audit_logs(created_at);
```

### 8.5 CAPTCHA Table

```sql
CREATE TABLE captcha_tokens (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    token_hash VARCHAR(255) NOT NULL UNIQUE,
    answer_hash VARCHAR(255) NOT NULL,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    used BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_captcha_expires ON captcha_tokens(expires_at);
```

## 9. API Design

### 9.1 Auth Routes

```
POST   /api/auth/login          — Login (username, password, captcha_token, captcha_answer)
POST   /api/auth/logout         — Logout
GET    /api/auth/me             — Get current user info
POST   /api/auth/captcha        — Generate new CAPTCHA
```

### 9.2 User Routes (authenticated)

```
GET    /api/users/profile       — Get own profile
PUT    /api/users/profile       — Update own profile
POST   /api/users/change-password — Change own password
```

### 9.3 Admin Routes (admin only)

```
GET    /api/admin/dashboard     — Dashboard stats
GET    /api/admin/users         — List users (search, pagination, filters)
POST   /api/admin/users         — Create user
PUT    /api/admin/users/:id     — Update user
DELETE /api/admin/users/:id     — Delete user (prevent last admin)
POST   /api/admin/users/:id/reset-password — Reset user password
POST   /api/admin/users/bulk-import — Bulk import CSV/TXT
GET    /api/admin/users/import-preview — Preview import file
```

### 9.4 Response Format

```json
{
  "success": true,
  "data": { ... },
  "message": "Success"
}

{
  "success": false,
  "error": "Error message",
  "code": "ERROR_CODE"
}
```

## 11. Panduan Deployment

### 11.1 Prerequisites

- Proxmox VE (latest)
- Ubuntu Server 22.04 LTS VM
- Docker + Docker Compose
- Domain name (internal)
- SSL certificates (Let's Encrypt atau internal CA)

### 11.2 Recommended Stack

**Ubuntu Server VM + Docker Compose**

Alasan:
- VM memberikan isolasi penuh
- Docker Compose memudahkan management
- Ubuntu LTS support lama
- Mudah di-backup dan restore
- Compatible dengan Proxmox

### 11.3 Deployment Steps

```bash
# 1. Clone repository
git clone https://github.com/ffigoperdana/igopdf.git
cd igopdf

# 2. Setup environment
cp .env.example .env
# Edit .env dengan konfigurasi

# 3. Build and start
docker compose -f docker-compose.yml up -d --build

# 4. Run migrations
docker compose exec backend npm run migrate

# 5. Seed database (development)
docker compose exec backend npm run seed

# 6. Check health
docker compose ps
curl http://localhost:80/health
```

### 11.4 Backup Strategy

- PostgreSQL: `pg_dump` harian
- Volumes: Proxmox backup
- Configuration: Git

### 11.5 Rollback Strategy

```bash
# Stop current deployment
docker compose down

# Checkout previous version
git checkout <previous-commit>

# Rebuild and restart
docker compose up -d --build

# Restore database if needed
docker compose exec backend npm run migrate:rollback
```

## 12. Panduan CI/CD

### 12.1 GitHub Actions

```yaml
name: Build and Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run lint
      - run: npm run test
      - run: npm run build

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:latest
          file: Dockerfile

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            cd /path/to/igopdf
            git pull
            docker compose pull
            docker compose up -d --remove-orphans
            docker compose exec backend npm run migrate
```

### 12.2 Migration Strategy

1. Jalankan migrations otomatis saat deploy
2. Gunakan versioned migrations
3. Test migrations di staging sebelum production

### 12.3 Rollback Strategy

1. Keep previous Docker images tagged
2. `docker compose` rollback ke previous image
3. Database rollback jika diperlukan
